Githome v0.1.1 is a performance release. The feature surface is the same as
v0.1.0: an unmodified `gh` CLI, stock `git`, GitHub-shaped REST v3 and GraphQL
v4. What changed is everything behind those responses. The list endpoints, the
search paths, the git object lookups, the GraphQL resolvers, the conditional
request fast path, the pkt-line writer, and the issue-create write path were all
reworked to hold their shape as the corpus grows. Nothing about the API
contract moves, so upgrading is a drop-in once the new migrations are applied.

This release also closes the full-text search gap that v0.1.0 shipped with:
search now runs on dialect-native indexes instead of LIKE scans.

## What changed

**List endpoints no longer fan out per row (P1).** Every list endpoint (issues,
pull requests, comments, check suites) used to issue one query per row for
labels, assignees, users, milestones, and reactions. A 30-row page cost 150 or
more queries. Seven batch loaders in `store/queries_batch.go` collapse that to
five or six queries per request regardless of page size, each a single
`WHERE pk IN (...)` with dialect-aware placeholder rebinding. Single-row paths
(GetIssue, GetPR, CreateIssue) are untouched, and a batch-vs-single consistency
test asserts the list view and the detail view cannot drift apart.

**Issue listing pages by seek, not offset (P2).** The default issue sort now
uses a `(created_at, number)` seek predicate instead of `LIMIT ? OFFSET ?`, so
fetching page N costs the same whether N is 1 or 1000 rather than scanning every
prior row. The cursor is `base64(unix_nano:number)` of the last item on the
page and rides in the `rel="next"` Link header. The random-access rels (`prev`,
`last`, `first`) keep page-number form, so `?page=N` still works for backward
navigation and total counts. A SQLite timestamp mismatch that made the seek
condition always true is fixed: `Store.timeArg()` now formats timestamps to
match the on-disk `YYYY-MM-DD HH:MM:SS` form for SQLite and passes `time.Time`
straight through for Postgres.

**Search runs on real indexes (P3).** `SearchIssues` and `SearchRepositories`
no longer scan with LIKE. SQLite gets FTS5 virtual tables (`issues_fts`,
`repos_fts`) declared with `content=` against the base tables so the text is not
duplicated, kept current by insert, delete, and update triggers, and backfilled
at migration time. Postgres gets `tsvector` columns generated and stored on
`issues` and `repositories`, indexed with GIN and maintained by the database.
Title-only and body-only queries still fall back to LIKE so the in-field
constraint is preserved, and the visibility predicate
(`private = false OR owner_pk = viewer`, plus `deleted_at IS NULL`) is unchanged.
Search never widens what a viewer can see.

**Git object lookups reuse processes and cache answers (P4).** `ObjectExists`
and `ObjectType` used to spawn `git cat-file -e` and `git cat-file -t` on every
call. They now route through a pool of long-lived `cat-file --batch-check`
processes, up to 64 of them with LRU eviction, one bound to each repository,
with every query to a repo serialized through its process. A dead process is
evicted and recreated transparently. In front of the pool sits a 4096-entry LRU
cache keyed by the 40-hex SHA; because a SHA is a content address, a cached
answer is valid forever, and blobs over 10 MB are left out to bound memory. The
common "check existence, then check type" pattern in ref updates now costs one
pool round trip instead of two spawns. The git Store also grows a `Close` that
drains the pool so the helper processes stop with the server.

**Database connection and cache tuning is configurable (P5).** Postgres pool
size is set by `GITHOME_DB_POOL_SIZE` (default 25), wired through to
`MaxOpenConns` and `MaxIdleConns`, so the pool is sized from a load test instead
of a hard-coded constant. SQLite gains a 64 MB page cache (`cache_size=-65536`)
and 256 MB of memory-mapped I/O (`mmap_size`), on top of the existing WAL,
`busy_timeout=5000`, and `synchronous=NORMAL`. Readers stop blocking on the
single writer and hot pages stay resident.

**GraphQL batches within a request and rejects abusive queries (P6).** A generic
per-request dataloader coalesces concurrent `Load` calls inside a 1 ms window
into one batch fetch, and a `Prime` method seeds the loader from already
assembled domain data so a query that touches both issues and pull requests
resolves their authors from cache instead of a second batch. The loaders wrap
the P1 store batch methods through a `domain.Batcher`, so the GraphQL layer never
imports the store. A complexity guard prices connection fields at
`first x childComplexity` the way GitHub does, caps a document at 5000, and
rejects anything nested deeper than 10 selection levels with a `MAX_QUERY_DEPTH`
error.

**Conditional GETs skip marshaling on the 304 path (P7).** For endpoints with a
stable version marker, a `304 Not Modified` no longer marshals the full body
just to hash it. `GET /repos/{owner}/{repo}` and
`GET /repos/{owner}/{repo}/issues/{number}` now derive their ETag from
`sha256(seed:id:updatedAt)`, which changes if and only if a rendered field
changes, and compare `If-None-Match` against it before touching the body. A
matching request returns 304 having paid only for the key. Endpoints without a
stable version marker keep the body-hash ETag.

**The pkt-line writer stops allocating (P8).** The Smart HTTP pkt-line encoder
replaced its `fmt.Fprintf(w, "%04x", n)` header and two-write pattern with a
`sync.Pool` byte buffer that holds the whole pkt-line and writes it in one call.
A benchmark confirms zero allocations per operation once the pool is warm. Two
tests lock the streaming invariant that the upload-pack response flushes after
every write so pack data reaches the git client incrementally, including the
graceful fallback for a writer that is not an `http.Flusher`.

**Issue creation does fewer write transactions (P9).** Creating an issue used
three write transactions: the main mutation, the event append, and the delivery
job enqueue. The event append and the `deliver_event` job now share one
transaction through `InsertEventAndJob`, taken by a fast path when the concrete
store supports it and falling back to the two-step path for test mocks. On
SQLite, which serializes all writers, that drops the create path from three
post-mutation writes to two; on Postgres it saves a round trip. The per-repo
number allocator already ran inside the main mutation with the minimum lock
span; a new test opens 20 issues concurrently and asserts the numbers come back
unique and contiguous from 1 to 20, so any race in the allocator is caught.
SQLite's single-writer ceiling is now documented in `store/store.go`.

## Upgrading from v0.1.0

The HTTP and GraphQL contract is unchanged, so existing clients need no changes.
Two operational notes:

- Run the migrations before starting the new binary. P3 adds the FTS5 tables and
  triggers on SQLite and the `tsvector` columns and GIN indexes on Postgres, and
  the migration backfills them from existing rows.

  ```sh
  githome-migrate up
  ```

- `GITHOME_DB_POOL_SIZE` is a new optional environment variable (default 25). It
  only affects Postgres. Leave it unset to keep the previous behavior.

## Install

Download the archive for your platform, verify the checksum, and put the binary
on your PATH:

```sh
# example: linux/amd64
tar -xzf githome_v0.1.1_linux_amd64.tar.gz
sudo mv githome_v0.1.1_linux_amd64/githome /usr/local/bin/
sudo mv githome_v0.1.1_linux_amd64/githome-migrate /usr/local/bin/
```

Or build from source with Go 1.26+:

```sh
go install github.com/tamnd/githome/cmd/githome@v0.1.1
go install github.com/tamnd/githome/cmd/githome-migrate@v0.1.1
```

## Quick start

```sh
# Postgres
export GITHOME_DATABASE_URL="postgres://user:pass@localhost/githome"
export GITHOME_HTML_URL="http://localhost:8080"
githome-migrate up
githome

# SQLite (no external database needed)
export GITHOME_DATABASE_URL="sqlite:githome.db"
githome-migrate up
githome
```

Point `gh` at it:

```sh
export GH_HOST=localhost:8080
gh auth login
gh repo create myorg/myrepo --private
git clone https://localhost:8080/myorg/myrepo.git
```

## Verify a release archive

```sh
sha256sum -c githome_v0.1.1_linux_amd64.tar.gz.sha256
```

## Platform support

Binaries are static (no libc dependency, no cgo) and built from a single
toolchain:

| OS      | Architectures                               |
|---------|---------------------------------------------|
| Linux   | amd64, arm64, arm, 386, ppc64le, s390x, riscv64, loong64 |
| macOS   | amd64, arm64                                |
| Windows | amd64, arm64, 386                           |
| FreeBSD | amd64, arm64                                |
| OpenBSD | amd64, arm64                                |

## Known gaps

Full-text search is now wired (it was the open search gap in v0.1.0). The
remaining deferred items are unchanged:

- SSH transport (clone and push over `git@host:owner/repo`)
- Org accounts (everything is user-owned for now)
- At-rest webhook secret encryption
- GitHub Actions runner compatibility
